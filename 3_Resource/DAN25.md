## 데이터 속 숨은 트렌드, 랭킹 기반 플레이스 트렌드 분석 시스템

- 지금 많이 찾는 장소 서비스 소개
    - 지금 뜨고 있는 장소는 어디일까?
    - 기존의 POI 중심의 주변 장소 추천에서 실시간 트렌드, 장소 컨텐츠 등 장소와 관련된 사용자의 니즈를 반영
    - 발견탭 출시에 맞춰 함께 출시
    - 체류시간은 358% 증가
    - 실시간성, 랭킹, 설명가능성 챌린지 존재
- 로그 속 진짜 신호 찾기
    - 검색은 관심, 클릭은 행동
    - 네이버지도는 일평균 70억건의 데이터를 생산
    - 업체별 순위를 알기 위해서는 업체별 검색, 클릭 수 집계가 필요
    - 10분 단위 집계를 통한 노이즈 최소화 및 의미있는 변화 포착
        - 안정성과 민감도를 균형있게 유지
    - 대규모 데이터를 실시간으로 효율적으로 집계하려면?
        - 다소 정합성이 떨어지더라도, 장기 지표를 빠르게 연산하기 위해 고민
        - 장기 집계용 데이터와 단기 집계용 데이터를 준비해두고 역할에 맞게 집계
- 랭킹 알고리즘과 보정
    - '급등'과 '지속' 을 함께 잡는 법
    - 모델링 핵심 아이디어
        - 시간의 무게
            - 단기부터 장기까지 여러 단계로 트렌드 변화 측정
        - 데이터의 목소리
            - 동적 조합
        - 품질의 문지기
- 렉스트 마이닝 x [[Large Language Models|LLM]]
    - 10분 안에 뜨는 이유를 키워드로 나타내려면?
    - 사용자들의 장소 데이터에서 키워드를 뽑아야 한다.
        - Transformer 기반의 LLM 으로 키워드 추출
    - 10분마다 하나의 장소당 1억7만 글자수, LLM 이 처리하려면 너무 많음
        - 게다가 품질을 위해 최소 200억 파라미터 크기의 모델을 사용해야 함
        - H100 을 사용하더라도 하나의 장소당 1일2시간이 걸린다는 계산
    - 입력 최소화가 필요
    - BERTopic
    - TextRank 알고리즘을 사용
        - 고전이지만 보편적인 모델
    - 기존의 텍스트랭크에서 biase 를 줘서 중요한 관련데이터만 뽑도록 유도함
    - Distillation 으로 LLM 경량화하기
    - Quantization
- 장소를 넘어 이슈로
    - 다른 컨텐츠와 POI 매핑 기술 필요
    - 어뷰징에 견고한 랭킹 설계

## 실시간 vs 배치, 함께 굴리는 데이터 파이프라인

## Observability  로 여는 새로운 가능성

- 로그 데이터 표준화 중요
- SLA = SLI + SLO
    - 매주 99% API 요청은 10ms 이하 라는 약속
- 세부적인 컴포넌트는 높은 신뢰성 확보를 위해 범용적인 오픈소스를 활용
- Json 형태 로그를 [[Kafka]] 를 통해 수집 및 적재
- 시계열 데이터베이스는 Victoria Metrics 사용
- Dashboard
    - Service health status
    - true positive alarm
    - percentile
    - SLI analysis chart
    - Service Level Objective
    - Variable Optional panel
- Threshold 기반 알람 & 급증/급감에 대한 알람 적용
- HTTP status code 50x 급증 현상 확인
- observability platform 프로젝트 경험기
    - Victoria metric 데이터 처리 아키텍처 및 주요 특징, 기능
        - 고성능 시계열 디비
        - [[Gorilla Algorithm]] 압축 방식으로 많은 데이터 저장 가능
        - 운영 편의성
        - 수평 확장 클러스터 지원
        - H/A 지원
        - PromQL 완벽 호환
            - [[Grafana]] 에서도 prometheus 데이터 소스 유형으로 연결해서 사용
    - [[Victoria Metrics]] 시나리오 테스트
        - 컴포넌트 장애 시나리오 테스트
    - 메트릭 카디널리티를 낮춰야 한다
        - 불필요한 필드는 최대한 제거
    - 데이터 간섭을 최소화하자
        - Tenant 분리
    - 알람 설정을 위한 차트 분석
        - 기본적으로는 threshold 기반 알람
        - 그러나 threshold 만으로는 정확하게 판단하기 어렵고 다각도로 분석 필요 및 설정이 필요
    - 장애감지나 리포트 생성을 자동화하기 위해 노력 중
- MCP Tool 과 새로운 분석 MCP Tool 을 사용하여 리포트 시스템 자동화 구축 중

어떻게 관측시스템을 구성했는지 알고 싶은데, 결과만 설명. [[Victoria Metrics]] 에 관해 집중 설명. [[InfluxDB]] 가 아닌 다른 시계열 데이터베이스를 찾고 있다면, 살펴볼만 해보인다.
